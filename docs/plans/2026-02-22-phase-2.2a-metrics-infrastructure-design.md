# Phase 2.2A: Baseline Metrics & Testing Infrastructure — Design Document

**Author**: Vivy (The Context Integrator)
**Date**: 2026-02-22
**Status**: Approved
**Master Plan Reference**: Phase II, Section 2.2A (Baseline Capture & Quick Wins)

---

## Problem Statement

The Omniscience memory system works but operates blind — no performance data is captured during recall operations. Without baseline metrics, we cannot measure whether future optimizations (2.2B: similarity filtering, deduplication, time-decay) actually improve anything. Additionally, there is no mechanism to enable/disable optimizations independently, making controlled comparison impossible.

Two open tasks from the master plan:
1. **Baseline Metrics Capture**: Document current memory system performance
2. **Testing Infrastructure**: A/B testing framework for memory optimizations

## Approach

**Integrated Metrics Layer (Approach A)** — instrument the existing recall pipeline directly rather than wrapping it with decorators. This is necessary because Vectra returns similarity scores that `recall()` currently discards, and we need those scores for quality metrics.

## Design

### 1. MemoryService Changes (`src/memory/memory-service.js`)

**1a. Return similarity scores from `recall()`**

Current return type:
```js
{ text, timestamp, source, chatId }
```

New return type:
```js
{ text, timestamp, source, chatId, score }
```

`score` is already available from Vectra's `r.score` in query results (line 96). Non-breaking change — no existing consumer reads `score`.

**1b. Fix dual memorization bug**

The user message is currently stored twice:
- Pre-generation: `chat-completions.js` line 2078
- Post-generation: `chat-completions.js` line ~2478

Remove the pre-generation call. Post-generation stores both user and assistant messages, which is the correct behavior.

### 2. MemoryMetrics Class (`src/memory/memory-metrics.js`)

New file. Lightweight instrumentation that captures per-request metrics and emits structured JSON.

**Captured metrics per recall:**

| Metric | Type | Description |
|--------|------|-------------|
| `latencyMs` | number | Wall-clock time for `recall()` call |
| `memoriesRetrieved` | number | Count returned from Vectra |
| `memoriesAfterBudget` | number | Count after token budget truncation |
| `similarityMin` | number | Lowest similarity score in results |
| `similarityMax` | number | Highest similarity score in results |
| `similarityMean` | number | Mean similarity score |
| `tokenBudgetUsed` | number | Tokens consumed |
| `tokenBudgetMax` | number | Token budget limit |
| `tokenBudgetExhausted` | boolean | Did we hit the cap |
| `characterName` | string | Which character |
| `featureFlags` | object | Snapshot of `omniscience.optimizations` |

**API:**
```js
const metrics = new MemoryMetrics(characterName, config);
metrics.startRecall();
const memories = await memoryService.recall(...);
metrics.endRecall(memories);
metrics.recordBudget(memoryLines.length, memories.length, tokensUsed, maxTokens);
metrics.emit(); // logs structured JSON
```

**Output destinations:**
- `logger.info('[METRICS]', JSON.stringify(payload))` — standard log infrastructure
- `{dataRoot}/omniscience_metrics.jsonl` — one JSON object per line, append-only, easy to analyze with scripts

**Guarded by**: `omniscience.metrics_enabled` config flag (default: `true`). When `false`, zero overhead — no metrics object created.

### 3. Feature Flags (`config.yaml`)

```yaml
omniscience:
  enabled: true
  recall_count: 5
  max_memory_tokens: 500
  embedding_model: "Xenova/all-MiniLM-L6-v2"
  metrics_enabled: true
  metrics_file: "omniscience_metrics.jsonl"
  optimizations:
    similarity_threshold: null       # 2.2B
    time_decay_days: null            # 2.2B
    deduplication: false             # 2.2B
    compression: false               # 2.2B
```

Each metric event includes a snapshot of the `optimizations` block. This creates a natural before/after comparison: run with all flags null/false to establish baseline, then enable one optimization at a time and compare metric distributions.

Accessed via existing `getConfigValue()` pattern — no new infrastructure needed.

### 4. Integration Point (`chat-completions.js`)

The Omniscience block (~lines 2048-2079) is modified to:
1. Create `MemoryMetrics` instance (if `metrics_enabled`)
2. Call `metrics.startRecall()` before `memoryService.recall()`
3. Call `metrics.endRecall(memories)` after — captures scores and latency
4. Pass score data through the token budget loop
5. Call `metrics.recordBudget()` after truncation
6. Call `metrics.emit()` at end of block

The existing control flow is preserved. Metrics calls are no-ops when disabled.

## What This Does NOT Do

- No similarity threshold filtering (2.2B)
- No deduplication (2.2B)
- No time-decay weighting (2.2B)
- No traffic splitting or automatic A/B assignment
- No dashboard or visualization
- No changes to the embedding model or vector storage

## Rollback Plan

- Set `omniscience.metrics_enabled: false` — disables all instrumentation
- Delete `omniscience_metrics.jsonl` — removes collected data
- Revert `recall()` score addition — non-breaking, just removes unused field
- Restore pre-generation memorize call if dual-memorization fix causes issues

## Success Criteria (from Master Plan)

- Baseline metrics documented and reproducible
- Metrics capture latency, similarity distribution, token efficiency per request
- Feature flags exist for all planned 2.2B optimizations
- Zero performance regression when metrics are disabled

## Relation to Master Plan

This completes the two remaining 2.2A tasks:
- **Baseline Metrics Capture**: MemoryMetrics class + JSONL output
- **Testing Infrastructure**: Feature flags + per-request flag snapshots enable before/after comparison

After this, Phase 2.2B can begin: each optimization enables its flag, and the metrics layer automatically captures the effect.
