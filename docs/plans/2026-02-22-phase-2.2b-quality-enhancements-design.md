## Problem Statement

The Omniscience recall pipeline returns raw top-K results from Vectra without quality filtering. This leads to:
- Low-similarity matches wasting token budget
- Near-duplicate memories consuming multiple slots
- Old memories treated equally to recent ones
- Long memories consuming disproportionate token budget

Phase 2.2A established the metrics infrastructure to measure these problems. Phase 2.2B adds four optimizations to address them, each independently toggleable via feature flags.

## Architecture

**Post-processing pipeline** — a single new file (`src/memory/memory-optimizer.js`) containing pure functions that transform the recall results between `recall()` and the token budget loop:

```
recall() → filterBySimilarity() → deduplicate() → applyTimeDecay() → compress() → [token budget loop]
```

Each step is a standalone function, conditionally executed based on its feature flag. The orchestrator function `optimizeMemories()` chains them together.

**Why pure functions over a class**: These are stateless transforms. Functions compose naturally, are easier to test individually, and avoid unnecessary abstraction.

## Design

### 1. Adaptive Similarity Filtering

**Function**: `filterBySimilarity(memories, threshold)`

Drops memories scoring below the configured similarity threshold.

**Edge case — memory desert**: If filtering would remove ALL memories, keep the single highest-scoring one. Users should always get at least one memory if Vectra returned results.

**Config**: `omniscience.optimizations.similarity_threshold: 0.3` (null = disabled)

**Metric**: `memoriesAfterFiltering` — count after threshold applied.

### 2. Intra-Memory Deduplication

**Function**: `deduplicate(memories, embeddingFn)`

Computes pairwise cosine similarity between memory texts using `getTransformersVector()`. When two memories exceed 85% similarity, the lower-scored one (by Vectra score) is dropped.

**Algorithm**:
1. Embed all memory texts via `getTransformersVector()`
2. Compute pairwise cosine similarity between embedding vectors
3. If similarity > 0.85, mark the lower-scored memory for removal
4. Return surviving memories in original order

**Performance**: For 5 memories, 5 embedding calls + 10 pairwise comparisons. At ~50ms per embedding, worst case ~250ms. Acceptable for a generation request.

**Config**: `omniscience.optimizations.deduplication: true` (false = disabled)

**Metric**: `memoriesDeduped` — count of memories removed.

### 3. Time-Decay Weighting

**Function**: `applyTimeDecay(memories, halfLifeDays, queryText)`

Re-scores memories by blending Vectra's similarity score with a recency boost using exponential decay:

```
decayFactor = 0.5 ^ (ageInDays / halfLifeDays)
adjustedScore = score * (0.7 + 0.3 * decayFactor)
```

The 70/30 blend ensures relevance dominates — a strong semantic match from months ago still beats a mediocre recent match. Between equally relevant memories, recent ones win.

**Query-awareness**: Checks query for atemporal keywords (`always`, `overall`, `history`, `everything`, `all time`, `ever`, `never`). If matched, decay is skipped entirely.

**After re-scoring**: Re-sort by `adjustedScore` descending. Original `score` preserved for metrics.

**Config**: `omniscience.optimizations.time_decay_days: 30` (null = disabled)

**Metric**: `timeDecayApplied` — boolean (false when atemporal query or flag disabled).

### 4. Memory Compression

**Function**: `compressMemories(memories, queryText, maxChars)`

Applies extractive compression to memories >200 characters using TF-IDF sentence scoring.

**Algorithm**:
1. Split text into sentences (regex: `/[.!?]+\s+/`)
2. If text <= 200 chars or only 1 sentence, skip
3. Build term frequency map from sentences + query
4. Score each sentence by sum of TF-IDF weights relative to query
5. Keep top-scoring sentences until total length reaches `maxChars` (default: 200)
6. Preserve original sentence order in output
7. Replace `memory.text` with compressed version

**No external dependencies** — TF-IDF implemented inline (~30 lines).

**Config**: `omniscience.optimizations.compression: true` (false = disabled)

**Metrics**: `memoriesCompressed` (count shortened), `compressionRatio` (mean compressed/original length).

### 5. Integration

**Orchestrator**: `optimizeMemories(memories, options)` exported from `memory-optimizer.js`. Runs each step conditionally based on flags.

**Call site in `chat-completions.js`**: Inserted after `recall()` and `metrics.endRecall()`, before the token budget loop.

**Metrics expansion**: `MemoryMetrics` gains a `recordOptimizations()` method capturing per-step effects. Called between the pipeline and the budget loop.

## What This Does NOT Do

- No adaptive K / dynamic recall count (2.2C)
- No query expansion (2.2C)
- No cross-system deduplication with personality state (2.2C)
- No changes to embedding model or vector storage
- No UI or user-facing commands

## Rollback Plan

- Each optimization has its own feature flag — disable individually
- Set all flags to null/false to return to baseline behavior
- Delete `memory-optimizer.js` and revert chat-completions.js wiring to fully remove
- Metrics continue capturing baseline data regardless

## Success Criteria (from Master Plan)

- Precision@K improves by 15-25% over baseline
- Diversity score improves (lower redundancy)
- Token efficiency improves by 20-30%
- Latency remains <100ms (excluding dedup embedding calls)
- User feedback confirms improved context quality

## Relation to Master Plan

This completes all four Phase 2.2B tasks:
- **Adaptive Similarity Filtering**: `filterBySimilarity()` with memory desert protection
- **Intra-Memory Deduplication**: `deduplicate()` with embedding-based pairwise comparison
- **Time-Decay Weighting**: `applyTimeDecay()` with query-aware keyword bypass
- **Memory Compression**: `compressMemories()` with TF-IDF sentence scoring

After this, Checkpoint 2A can be evaluated (requires 2.2A + 2.2B complete, 20%+ metric improvement).
